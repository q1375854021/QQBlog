Lab3-Page Table 笔记

本实验主要围绕 RISC-V 的分页机制展开，深入探讨虚拟地址到物理地址的转换过程、页表的遍历与打印、以及超级页（Superpage）的应用。通过这些实验，我们可以更好地理解操作系统如何利用硬件分页机制实现内存的虚拟化。

说明：本实验基于 XV6-2024 版本，实验指导书地址：https://pdos.csail.mit.edu/6.1810/2024/labs/syscall.html

(1) 虚拟地址 (VA)

va 0xFFFFF000 是 RISC-V Sv39 模式下的 39 位虚拟地址。

(2) 页表项 (PTE)

pte 0x2000184B 是该虚拟地址对应的页表项值。这个值是如何得到的呢？

在 RISC-V Sv39 模式下，虚拟地址被划分为以下四个部分：

字段 位范围 含义
L2 (VPN[2]) [38:30] 二级页表索引，9 位
L1 (VPN[1]) [29:21] 一级页表索引，9 位
L0 (VPN[0]) [20:12] 零级页表索引，9 位
Offset [11:0] 页内偏移，12 位 (4KB)

页表遍历过程详解：

pgtbltest 是一个用户进程，拥有用户页表（两级页表结构）。无论是 MMU 硬件还是软件访问页表，都会遵循以下流程：

  1. 第一级索引：读取根页表 pagetable[L2]，得到 PTE2。

    • PTE2[53:10] = PPN2（物理页号）
    • PTE2[9:0] = Flags（权限标志）
    • 通过 PTE2 去除 Flags，然后 (PTE2 >> 10) << 12，获取一级页表的物理地址。

  2. 第二级索引：读取一级页表 pagetable[L1]，得到 PTE1。

    • PTE1[53:10] = PPN1
    • PTE1[9:0] = Flags
    • 通过 (PTE1 >> 10) << 12，获取零级页表的物理地址。

  3. 第三级索引：读取零级页表 pagetable[L0]，得到 PTE0。

    • PTE0[53:10] = PPN0
    • PTE0[9:0] = Flags
    • 获得 PTE0 后，计算物理地址：PA = ((PTE0 >> 10) << 12) | VA[11:0]
    • 其中 (PTE0 >> 10) << 12 是物理页的起始地址，Flags 是页属性。

(3) PTE Flags 详解

属性字段是 Flags[9:0]，各位含义如下：

位 标志 名称 说明
bit[0] V Valid 页表有效位。为 0 表示当前虚拟地址未分配物理内存
bit[1] R Readable 1 表示允许读取
bit[2] W Writable 1 表示允许写入
bit[3] X Executable 1 表示该页可作为机器指令执行
bit[4] U User 用户态访问位。为 0 表示仅 S Mode（内核态）可访问
bit[5] G Global 全局位。G=1 表示全局映射，切换进程时不刷新 TLB
bit[6] A Accessed 访问位。CPU 读或写过此页时会自动置 1，OS 可据此判断冷热数据
bit[7] D Dirty 脏位。CPU 写过此页数据时自动置 1。OS 回收页面时，若 D=0 可直接丢弃，D=1 必须写回 swap
bit[9:8] - Reserved 保留位，供超级软件使用

(4) 权限位解析

perm = 0x4B = 0b01001011，各位解析如下：

位 值 含义
bit[7] D 0 未被修改
bit[6] A 1 已被访问
bit[5] G 0 非全局
bit[4] U 0 仅内核态可访问
bit[3] X 1 可执行
bit[2] W 0 不可写
bit[1] R 1 可读
bit[0] V 1 有效

因此，这是一个仅可执行的内核页，这正是 XV6 中用于跳转到用户态执行代码的 Trampoline 页面。

答案分析：

任何满足以下条件的系统调用都可以使用这个共享页加速：

  1. 只读取某个数据：安全可靠，如获取进程信息、时间等。
  2. 仅修改某个变量：可行但有一定风险，通常需要特殊处理。
  3. 实现简单：最好只涉及一个页的数据，避免调用其他分散的内核函数。

为什么限制在一个页内？

因为内核函数的地址分布很分散，如果用户程序需要访问多个不同位置的数据，就需要映射更多页，这会增加安全风险（除非一次性映射所有页，但这极其危险）。

根据这个原则，uptime 系统调用也可以使用共享页加速。uptime 用于获取系统运行时间，只需在共享页中存放一个时间值，用户直接读取即可，无需陷入内核。

4. Use Superpages

实验背景

在标准的分页机制中，物理内存被切成 4KB 的小块。虽然这提供了灵活的内存管理，但也有缺点：TLB（Translation Lookaside Buffer，页表缓存）的容量有限。TLB 是 CPU 内部一个极其昂贵且容量极小的硬件缓存，如果全是 4KB 小页，TLB 很快就装满了，一旦 TLB Miss，就必须去内存中缓慢查表。

超级页（Superpage）一般是 2MB，一条 TLB 记录就能覆盖 2MB 空间。对于内存密集型程序，这将极大提升性能，显著降低 TLB Miss 的概率。

实现超级页的关键在于：

  1. 区分超级页和普通页：通过 PTE_LEAF 宏判断 PTE 是否为叶子节点。如果当前处于 L1 级页表且 PTE 是叶子节点，则它是一个超级页。

背景知识

1. SV39 Virtual-Memory System

详细可参考 RISC-V 特权架构手册的第 12.4 节。

(1) 虚拟地址结构

  • 总宽度：39 位（实际使用 bit[37:0]，即 38 位）
  • 划分方式：9bit VPN[2] + 9bit VPN[1] + 9bit VPN[0] + 12bit Page Offset
  • 总寻址空间：2^38 = 256GB（由于地址扩展规则，放弃最高位）

(2) 物理地址结构

  • 总宽度：56 位
  • 划分方式：26bit PPN[2] + 9bit PPN[1] + 9bit PPN[0] + 12bit Page Offset
  • 总寻址空间：2^56 = 64PB

(3) PTE 结构

  • 总宽度：64 位
  • 包含：物理页号（PPN）、权限标志位（Flags）等

为什么 XV6 使用 38 位虚拟地址？

指令取址和加载有效地址都是 64 位的，其中 bit[63:39] 与 bit[38] 必须保持一致（即符号扩展）。为避免处理这种复杂的边界情况，XV6 将地址空间限制为 bit[37:0]，即 38 位。因此 MAXVA = 1 << 38，地址空间从 $2^{39} = 512GB$ 降低到了 $2^{38} = 256GB$。

2. 关于 Trampoline

参照 kernel.ld，链接器在 KERNBASE = 0x80000000 上放置 text 段后，经过 4KB 对齐放置了 trampoline。参照 XV6 Book 的 Figure 3.3，右侧物理地址在 KERNBASE 上面一点放置了 Trampoline，左侧虚拟地址空间的 TRAMPOLINE（MAXVA - PGSIZE）正好映射到这个位置，实现了对应关系。

Trampoline 的作用：它是一个代码页，用于在用户态和内核态之间切换时进行上下文保存和恢复。由于它需要在两个地址空间中都可见，因此被映射到了固定位置。

3. 虚拟地址和物理地址如何映射？

参照 XV6 Book 的 Figure 3.3，有很多部分是直接映射的（Virtual Address = Physical Address）。这是如何实现的？

其实质就是通过页表中的 PTE 来建立映射关系。可以参照 vm.c 的 mappages 函数：

映射原理：

  1. 确定虚拟地址范围（起始和终止地址）
  2. 找到虚拟地址对应的 PTE，将 PTE 设置为 物理地址 | 属性
  3. 循环处理，直到所有页都完成映射

关键点：

  • 如果多个 VA 的页映射到同一个 PA，称为共享内存（Shared Memory）。在任意一个 VA 上修改数据，其他 VA 也能看到变化（mmap、共享库、IPC 等都依赖此特性）
  • 注意：VA 只能释放一次。如果所有 VA 都释放了才去释放 PA，会导致内核 panic（因为 XV6 内核没有引用计数）

直接映射的实现：传入相同的 VA 和 PA 值即可。例如 kvmmap(kpgtbl, KERNBASE, KERNBASE, size, flags) 将内核地址空间直接映射到相同的物理地址。

4. 内核的 VA 和 PA 是如何映射的？

参照 vm.c 中的 kvmmake 函数，kpgtbl 是内核页表的基址。内核的地址映射包括：

(1) UART0：直接映射，一个 PGSIZE（4KB）

(2) VIRTIO0：直接映射，一个 PGSIZE

(3) PLIC：直接映射，64MB。为什么这么大？为了支持超大规模的服务器（如主板上有几千个 CPU 核心、几万个外设），PLIC 的控制寄存器被设计得极其稀疏和庞大。而且每个 CPU 核心的 M 模式、S 模式都需要挂起、使能等寄存器。

(4) KERNBASE：直接映射，这是内核代码段，可直接执行。映射大小为 etext - KERNBASE，具体可查看 kernel.ld。

(5) etext：直接映射，大小为 128MB - 内核text大小。etext 包括内核定义的全局变量和静态变量，以及所有可用 RAM。这段内存可读写但不可执行，因为 128MB 内存存放了操作系统的完整映像。

(6) trampoline：不是直接映射。它将虚拟地址 TRAMPOLINE（MAXVA - PGSIZE）映射到物理地址的 trampoline 上。因此在内核中访问 trampoline 必须通过 TRAMPOLINE 虚拟地址，而不能直接用物理地址。

(7) proc_mapstacks：为每个进程分配内核栈：

这里只是为每个进程分配好了栈空间，栈的内容是空的。

为什么每个进程都需要一个内核栈？

当用户程序调用 write() 或发生中断时，CPU 会陷入内核态。内核态执行的是 C 语言代码，只要是 C 语言函数调用，就离不开栈（用于存储局部变量、函数返回地址等）。

内核绝对不敢使用用户程序自己的栈（User Stack），因为恶意用户可能故意破坏用户栈。因此，当进程切入内核态时，必须立刻切换到一个完全属于内核、干净且安全的内核栈来运行。这就是 kalloc 申请的那一页物理内存的作用。

5. 什么是 Super Page（超级页）？如何区分普通页表与超级页？

在标准内存分页机制中，物理内存被切成极小的块，最常见的大小是 4KB。由于 CPU 内部的 TLB（页表缓存）容量极其有限，如果全是 4KB 小页，TLB 很快就装满了，一旦 TLB Miss，就必须去内存中缓慢查表，性能损失严重。

超级页一般是 2MB，TLB 里一条记录就能覆盖 2MB 空间。对于内存密集型程序，这将极大提升性能，大幅降低 TLB Miss 的概率。

普通页（4KB）和超级页（2MB）是如何得来的？

在 Rv-39 模式中，虚拟地址结构为 9bit(L2) + 9bit(L1) + 9bit(L0) + 12bit(offset)：

  • 对于 4KB 页：只有 12bit offset 用于页内偏移，前面的 L2、L1、L0 都用于索引页表
  • 对于 2MB 超级页：L0 + offset（18bit）组成新的 offset（2^21 = 2MB），L0 级别变成了超级页的物理页号，而不需要再索引下一级

如何区分当前 PTE 是指向超级页还是下一级 4KB 页表？

这是一个关键问题，想清楚这个问题，超级页实验就完成了一半。

答案：通过 Flags 来区分！

参照 XV6 Book 的 Figure 3.2，假设当前处于 L1 页表，pte = &pagetable[L1]。有以下情况：

  1. V 位无效：说明这个 PTE 还未分配。分配内存时 V 无效没问题（可以分配），但索引内存时 V 无效就会 panic（表明索引错误）。

  2. 在 V 位有效的前提下：使用 PTE_LEAF 判断当前 PTE 是指向 L0 页表还是物理页帧：

    • 如果指向 L0 页表：Flags 中 RWX 必然全为 0（因为没有读写执行权限）
    • 如果指向 超级页的物理页帧：Flags 中 RWX 至少有一个为 1

这就是 walk 函数中 PTE_LEAF 判断的原理。

为什么 epc 是 0？

因为 uvmfirst 中将虚拟地址 0 映射到了 initcode（第一个程序）的代码段，所以 PC 设置为 0 是正确的。栈地址设置为 PGSIZE，足够使用。

然后 userinit 返回到 main 函数。main 函数执行到 scheduler，scheduler 执行 swtch(&c->context, &p->context)。由于 allocproc 之前将 p->context.ra 设置为 forkret，所以调度器直接切换到了 forkret。

forkret 执行 usertrapret 函数：

最后执行 userret，恢复所有寄存器，然后执行 sret 指令。sret 会：

  1. 将 CPU 特权级从 S Mode 降到 U Mode
  2. 将 PC 设置为 sepc 寄存器的值
  3. 打开中断

PC 指向 0，即 initcode，执行 exec("/init", {init, 0})，这会调用 syscall，跳转到 trampoline。

trampoline 系列 syscall 调用流程

  1. 保存所有寄存器到 trapframe 中
  2. 提取关键参数：kernel_sp（进程内核栈）、kernel_hartid（CPU ID）、kernel_trap（usertrap 地址）、kernel_satp（内核 satp）、epc（ecall 发生时的 PC）
  3. 切换页表，跳转到 usertrap
  4. usertrap 先将 kernelvec 保存到 stvec（因为已进入内核，中断处理函数要切换）
  5. 保存 epc 寄存器到 trapframe（epc 正是 syscall 发生时刻的 PC 值）
  6. epc += 4（返回地址跳到 ecall 的下一条指令）
  7. 调用 syscall：通过 a7 确定 syscall 类型，调用对应的处理函数
  8. sys_exec 调用 exec，完成进程内存空间构建。关键是设置 p->trapframe->epc = elf.entry
    • 普通 syscall 不会改变 epc，只有 exec 特殊（因为它不会返回到原程序）
  9. 返回到 usertrap，执行 usertrapret
  10. usertrapret 更新五个参数，然后将 epc 写入 sepc
  11. 跳转到 trampoline 的 userret，写入 satp，恢复寄存器，执行 sret
  12. sret 将 PC 设置为 sepc，即 init 的 main 函数地址

为什么说 sys_exec 很特殊？

传统 syscall 会返回到 ecall 的下一条指令，但 exec 会跳转到新程序入口点。

特殊的 fork 系统调用

  1. fork 肯定是 syscall，从用户态跳转到 trampoline
  2. 将 a0 保存到 sscratch，然后 a0 指向 TRAPFRAME
  3. 备份 proc 的 trapframe，提取 kernel_sp、kernel_satp 等参数，切换页表，进入 usertrap
  4. fork 中有 *(np->trapframe) = *(p->trapframe)，子进程的 trapframe 与父进程相同
  5. 返回到 usertrap，执行 usertrapret，更新参数，分别切换到各自进程的页表
  6. userret 恢复寄存器，sret 将 PC 设置为 ecall 的下一条指令

虽然 fork 执行了一次，但由于创建了子进程和新的地址空间，实际上返回了两次（父进程返回子进程 PID，子进程返回 0）。

7. swtch 和 trampoline 均涉及进程切换，为什么 swtch 保存/恢复的寄存器比 trampoline 少？

根本原因：swtch 是函数调用，trap 是突发的。

swtch 调用：swtch(&c->context, &p->context)

这是函数调用，受 RISC-V C 语言调用约定的约束。在 RISC-V 的 32 个通用寄存器中：

  • Caller-Saved 寄存器：临时寄存器（t0-t7）和参数寄存器（a0-a7）。这些在函数调用时会被调用者保存到栈上。
  • Callee-Saved 寄存器：保存寄存器（s0-s11）、栈指针（sp）和返回地址（ra）。这些由被调用者负责保存和恢复。

为什么平时写 C 语言时没有处理过 Callee-Saved 寄存器？

因为编译器会在函数调用前后自动生成代码来处理。但如果是写汇编，就需要手动操作了。

Trap 是突发的：没有提前准备，因此需要保存所有 32 个通用寄存器到 trapframe 中。

8. 0, Sv39, Sv48, Sv57

通过设置 satp 寄存器可以切换不同的页表模式：

模式 含义 地址结构 寻址空间
0 关闭 MMU，直接映射 - -
Sv39 Supervisor + Virtual + 39 9+9+9+12 512GB
Sv48 Supervisor + Virtual + 48 9+9+9+9+12 256TB
Sv57 Supervisor + Virtual + 57 9+9+9+9+9+12 128PB

为什么需要这些不同的模式？

优势：

  1. 大型数据中心可能需要几十 TB 的数据映射，需要更大的寻址空间
  2. 大内存更容易实现 ASLR（地址空间布局随机化），使代码、堆、栈分布更远，增加黑客攻击难度

劣势：

  1. TLB Miss 代价变大：Sv57 需要访存 5 次（五级页表）
  2. 页表本身占用更多内存

9. XV6 的空闲物理页是怎么管理的？

main 函数调用 kinit，kinit 调用 freerange(end, (void *)PHYSTOP)，freerange 对每个页调用 kfree 进行初始化。

kfree 是管理空闲物理页的核心：

struct run 是一个单链表节点（只有 next 指针，没有 value）。

头插法：新的空闲页插入到链表头部，因此 kmem.freelist 始终指向最近释放的页。LIFO（后入先出）的策略保证了 Cache 命中率更高——最近释放的页更可能还在 CPU Cache 中。

问题：物理地址释放后，虚拟地址也跟随失效。Cache 存放的是虚拟地址，为什么说 LIFO 的 Cache 命中率更高？

解析：CPU Cache 的工作方式：

  • L1 Cache：通常采用 VIPT（Virtually Indexed, Physically Tagged）

    • 查找时（Index）：使用虚拟地址的一部分快速定位 Cache 行（使用低 12 位，找到可能的几个 Tag）
    • 确认时（Tag）：必须比对物理地址。MMU 同时将虚拟地址翻译成物理地址，只有两者完全一致才算命中

  • L2/L3 Cache：绝大多数采用 PIPT（Physically Indexed, Physically Tagged），完全使用物理地址

为什么 LIFO 仍然有效？

虽然 L1 Cache 使用虚拟地址索引，但确认时需要物理地址 Tag。而且现代 CPU 通常有复杂的 Cache 预取机制，LIFO 释放的页往往在物理上是连续的，更容易被 Cache 预取命中。
