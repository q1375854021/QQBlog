1. Boot xv6(easy)

1. make qemu 报错：Couldn’t find a riscv64 version of GCC/binutils

错误现象：
执行 make qemu 时提示找不到 riscv64 架构的 GCC 编译器和 binutils 工具链。

原因分析：
XV6 操作系统是为 RISC-V 64 位架构编写的，因此需要专门的交叉编译工具链。交叉编译是指在一个架构（如 x86_64）上编译生成另一个架构（如 riscv64）的可执行文件。普通的 x86 编译器无法生成 RISC-V 指令集的代码。

我的推测： 是因为我的环境里面没有安装 riscv64 的 GCC 编译工具链。（推测正确）

解决方案：

各组件作用说明：

组件 作用
gcc-riscv64-linux-gnu RISC-V 64 位交叉编译器，将 C 代码编译为 RISC-V 机器码
binutils-riscv64-linux-gnu RISC-V 二进制工具集（as、ld、objdump 等），用于汇编、链接和目标文件分析
qemu-system-misc QEMU 模拟器，支持 RISC-V 架构的系统级模拟
gdb-multiarch 支持多架构的 GDB 调试器，可用于调试 RISC-V 程序

验证安装：

执行上述命令后显示版本信息即表示安装成功。

结果： 安装完成后，make qemu 编译成功。

2. 编译卡死：没有出现 $ 符号

错误现象：
编译虽然成功，但 QEMU 启动后一直卡在某个界面，始终没有出现预期的 shell 提示符 $。

排查思路：
这是一个比较难定位的问题，需要采用排除法逐步缩小范围。

(1) make clean 再编译

  • 目的：排除之前编译残留文件导致的问题
  • 结果：同样结果，问题依旧

(2) make clean, make CPUS=1 qemu，设定为单核启动

  • 目的：排除多核并发导致的竞争条件问题
  • 结果：同样结果，问题依旧

(3) 怀疑是编译版本的问题

  • 排查了 Makefile 脚本，以及工具位置
  • 结果：配置没有问题

关键线索：
只能按 Ctrl+A 然后按 X 退出 QEMU，退出时打印 QEMU: Terminated。这说明 QEMU 环境已经成功启动，但操作系统没有完成 boot 过程。

深入排查 - 使用 GDB 调试：

使用 make qemu-gdb 启动带 GDB 调试的 QEMU，然后连接 GDB 进行排查。

发现问题：
在 timerinit 函数中的 w_mie(r_mie() | MIE_STIE); 这行代码处，程序跳转到了地址 0 处。

初步推断： timer 的中断向量表没有正确设置。

进一步发现：
XV6 官方网页目录的 Labs 中有 tools 和 guidance 文档，之前没有仔细阅读，可能存在环境配置上的遗漏。具体见《Tools.md》。

最终解决：
按照《Tools.md》配置好环境之后，make qemu 就成功了。

核心问题解答：

Q: 我的代码里确实 timervec 没有设置，为什么配置了环境就可以了呢？

A: 这是 QEMU 版本与 RISC-V 扩展兼容性的问题。

详细解释：

  1. sstc 扩展是什么？

    sstc (Supervisor Software Timer Counter) 是 RISC-V 的一个扩展，允许 S 模式（监督模式）直接访问时间计数器。2024 年的 XV6 代码用到了 RISC-V 的 sstc 扩展来实现时钟中断。

  2. QEMU 6.2.0 的问题：

    QEMU 6.2.0 版本不支持 sstc 扩展，而新版 XV6 代码依赖这个扩展。当 QEMU 不支持所需的 RISC-V 扩展时，会导致时钟中断机制无法正常工作，系统无法完成初始化。

  3. 为什么升级 QEMU 后就好了？

    QEMU 7.2.0 及以上版本支持 sstc 扩展，升级后 XV6 可以正常使用新的时钟中断机制。

  4. 当前 XV6 的时钟中断流程：

    在配置好环境后，时钟中断的配置不在 timervec 中，而是在 devintr 函数中处理。具体跳转过程如下：

寄存器 作用
stvec 监督模式陷阱向量基地址寄存器，定义异常/中断的入口地址
scause 监督模式异常原因寄存器，0x8000000000000005 表示 S 模式定时器中断
sie/mie 中断使能寄存器，MIE_STIE 位启用软件定时器中断

经验总结：

  • 遇到 XV6 问题时，先仔细阅读官方 Labs 的 tools 和 guidance 文档
  • QEMU 版本对 RISC-V 扩展的支持很重要，建议使用官方推荐的 7.2.0 版本
  • 使用 GDB 调试可以帮助定位底层硬件相关的问题

2. sleep(easy)

实验目标： 实现系统调用 sleep()，使进程可以挂起指定的时间片。

涉及文件：

  • kernel/sysproc.c - 添加 sys_sleep() 系统调用
  • kernel/proc.c - 理解 sleep() 和 wakeup() 机制
1. sys_sleep 的 sleep(&ticks, &tickslock) 实现

实现原理：

(1) 进入睡眠状态： 当前进程 p 转变为 Sleeping 态，然后将 p 的等待频道 p->chan 指向 ticks。chan 是一个通道指针，用于标识一组等待同一事件的进程。

(2) 全局时钟计数器： ticks 是全局的时钟计数器，每次时钟中断，ticks 都会加 1，并且使用 wakeup(ticks) 把在 ticks 通道上所有睡眠的线程唤醒。

(3) 循环检查： 然后回到 sys_sleep 中，检查是否需要真正唤醒了，如果还没有到唤醒时间，再次执行 sleep。

(4) 惊群效应： 因为每个睡眠的线程都会唤醒，每个线程都进行检查，所以非常消耗时间（惊群效应）。因为 XV6 只是简单的教学操作系统，现代的操作系统都会用定时器红黑树、时间轮来计算谁先醒来。

(5) 效率问题： 效率极低，每个睡眠的线程都会经历一个过程：从调度器切换到该线程 → 检查 sys_sleep 的 while 循环发现时间没到 → 再次调用 sleep → 立刻切换回调度器。假如有 100 个睡眠线程，那么 CPU 会进行 99 次无意义的上下文切换。

知识点扩展：

sleep() 和 wakeup() 机制：

现代操作系统的优化：

技术 原理 优势
定时器红黑树 按超时时间排序，O(log n) 查找 快速找到最近的定时器
时间轮算法 类似时钟的循环数组 O(1) 插入和删除
分级时间轮 多层时间轮处理不同精度 支持长延时和高精度

思考题：

  1. 为什么 sleep() 需要先 release(lk) 再 sched()？
    • 答：避免死锁，让其他进程有机会获取锁并调用 wakeup()
  2. 为什么被唤醒后还要重新 acquire(lk)？
    • 答：保证调用者期望的锁状态，维持不变式
2. 时钟中断频率

(1) clockintr() 函数内有着这样一个语句：w_stimecmp(r_time() + 1000000);

  • r_time() 是操作 time 寄存器（时钟计数器，CPU 时钟频率的计数器，由晶振 +PLL 产生，通常不是 RISC-V 基金会考虑，而是 CPU 厂商考虑）。time 是 0xC01，非特权级手册有写。
  • w_stimecmp() 操作 stimecmp 寄存器 (地址 0x14)：RISC-V 特权级手册有写。
