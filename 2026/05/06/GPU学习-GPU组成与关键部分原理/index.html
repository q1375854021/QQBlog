GPU组成与关键部分原理

了解CUDA编程怎么可以不先了解GPU的组成和内部原理呢？了解了GPU的底层构造之后，对CUDA代码进行调优才更能得心应手。

GPU DRAM

现代GPU通常使用特殊的显存颗粒：如GDDR（目前最新为GDDR7）或HBM。

GDDR

相比于DDR追求低延迟，GDDR追求高吞吐量。

为什么GDDR能够实现超高带宽？

GDDR的接口位宽通常比DDR宽得多。通过并行连接多个GDDR颗粒扩展数据总线宽度（参照微机原理的RAM扩展），从而形成了256bit甚至384bit的总线宽度。

除了数据总线并行扩展，每个GDDR芯片内部通常被划分为两个独立的16bit通道，这意味着可以同时处理两个不同的内存请求了。

RTX4060是GDDR6，只有128bit显存带宽，4个显存颗粒。而CPU内存双通道也可达到128bit内存带宽。主要是GDDR用了QDR（SDR是1倍，DDR是2倍，QDR是4倍），这是因为GDDR通常焊在GPU周围，走线约束什么的比较好做。但是DDR通常是DIMM的，很难约束QDR，因此基本上都是DDR。

当然，CPU目前也借鉴了GDDR的思想，PS5就把一整圈GDDR6颗粒焊在了APU（CPU+GPU）周围。苹果的M2/M3/M4把内存颗粒和CPU焊在了同个芯片内，直接得到了极高的带宽。

不同代的GDDR的区别是什么？

GDDR1（基于DDR）：其实就是早期的高频版DDR。显卡厂家发现DDR挺好用，就挑体质好的拿来当显存。

GDDR2（基于DDR2）：这是一个著名的”翻车”点。它盲目追求高频率，导致发热极其恐怖（当年的NVIDIA FX 5800 Ultra显卡因此被称为”吹风机”），寿命极短，很快就被淘汰。

GDDR3（一代经典）：虽然它叫3，但技术依然基于DDR2。工程师吸取了教训，大幅降低了工作电压，并专门为GPU优化了读写延迟。这是第一款真正意义上为GPU“量身定制”的显存，极其成功，统治了市场好几年。

GDDR4（昙花一现）：它的预取翻倍了（一次能拉回更多数据），并引入了DBI（Data Bus Inversion，数据总线反转）技术来降低高频功耗。但因为它成本高、延迟大，NVIDIA直接跳过了这一代，只有AMD用过一小阵子。

GDDR5（绝对的传奇）：统治了整整一个时代的王者。从这一代开始，GDDR引入了双时钟系统，这让它的等效频率轻松突破了8Gbps。

GDDR5X（过渡期的黑科技）：首次引入了QDR（四倍数据速率）的概念。在一个时钟周期内传输4次数据，让GTX 1080的带宽暴涨。

GDDR6：物理通道切分！它把原本一颗芯片的单32-bit通道，硬生生劈成了两个独立的16-bit通道，极大地提高了访存效率。

GDDR6X：GDDR6X突破性地使用了PAM4（四电平脉冲幅度调制）。它有4种电压状态，意味着它一个时钟周期能传输2个bit的数据（00, 01, 10, 11）（如RTX 4090）。

GDDR7：GDDR7将一颗物理芯片硬生生切分成了4个独立的通道，采用3种电平状态。

HBM（High Bandwidth Memory）

HBM是一种高端显存技术，与GDDR有着本质的不同。HBM通过TSV（Silicon Through Via，硅通孔）技术将多层DRAM芯片堆叠在一起，并与GPU芯片通过硅中介层（Silicon Interposer）垂直连接。

HBM的技术优势：

  • 极致带宽：HBM2可以达到256GB/s甚至更高的带宽（如H100 SXM的80GB版本使用HBM3，带宽超过3.35TB/s）
  • 超低功耗：由于连接距离极短，信号衰减和功耗都大大降低
  • 小尺寸：3D堆叠结构大大减小了显存占用的PCB面积
  • 高密度：单颗HBM颗粒可以实现更大的显存容量

HBM的局限：

  • 成本极高：3D堆叠和硅中介层工艺复杂，成本是GDDR的数倍
  • 无法升级：只能与特定GPU封装在一起，无法独立升级
  • 良品率较低：产能受限

HBM的应用场景：HBM主要应用于数据中心GPU（如NVIDIA H100、A100、AMD MI250等）和高端专业显卡，因为这些场景对带宽和功耗的需求远超消费级市场。

NVLink与NVSwitch

NVLink是NVIDIA开发的高速互联技术，用于GPU与GPU之间或GPU与CPU之间的数据传输。相比PCIe，NVLink提供了更高的带宽和更低的延迟。

NVLink的技术特点：

  • 每个NVLink链路提供双向25GB/s的带宽（第一代）到50GB/s（第三代）
  • 一个GPU最多可以支持18个NVLink链路（如H100）
  • 延迟比PCIe低很多，因为NVLink绕过PCIe协议栈直接通信

NVSwitch：当需要连接多个GPU时，NVSwitch提供了全连接的Crossbar拓扑结构。在NVLink的基础上，NVSwitch允许任意GPU直接与其他任意GPU通信，不会产生通信瓶颈。例如DGX H100系统中，8个H100 GPU通过NVSwitch全互联，实现任意GPU之间900GB/s的通信带宽。

NVLink的应用：在深度学习训练中，多GPU协同工作需要频繁交换梯度数据。NVLink使得这种数据交换几乎不占用PCIe带宽，让GPU可以专注于计算。在实际测试中，8卡A100通过NVLink互联的训练速度是8卡通过PCIe互联的2-3倍。

显存控制器Memory Controller

MC是物理分割成了多个独立的显存分区，每个分区拥有自己独立的映射器、合并引擎和调度队列。RTX4060有四个显存颗粒，所有MC分成了四份，四个MC。每个MC负责管理一个显存颗粒芯片。

L2 Cache也被切为了4块。一块L2 Cache对应一个MC对应一个显存颗粒。

MC的工作原理

1，请求数据

当向MC请求数据时，MC读取GDDR的一整行数据放在MC内部的Row Buffer中，如果接下来的几十个线程读取的数据刚好都在这一行，那么线程直接从Row Buffer读取数据就可以了，就不用再访问显存了。

2，管理优先级

屏幕刷新是最高优先级，当屏幕刷新信号到了，MC会暂停所有的CUDA显存请求，把带宽全都让给显示输出。对于同级别的CUDA请求，MC会先找到请求队列里的”Row Buffer命中”的请求，让它插队。

3，管理GDDR的动态刷新

4，不断微调电压幅度与时钟延迟，直到眼图最佳

也就是说就算是随机访问数据，只要访问请求能均分到不同的MC上，并且每个MC内的Row Buffer能最大限度命中，那么显存利用率就能达到最高。不一定要完全线性合并访问。

当然，随机访问数据时，访问请求大概率能够均分到所有MC上，但是行命中很难保证，所以随机访问的开销在于行命中上。并且由于随机访问，通常只用某4个字节，但是MC从GDDR里取的数据通常是32字节或者128字节（取决于L2 Cache Line的大小）。也就是说当取某4个字节，命中MC1，但由于一次读取一个L2 Cache Line，所以M0、M2、M3也会强制去请求对应的显存颗粒，导致每个MC都承担了75%的无效工作。但是一次读取一个Cache Line是CPU思维，GPU在128字节的Cache Line里划分了4个32字节的Sector，假如某4个字节命中了MC0，那么只有MC0会读取这4个字节，MC1、2、3都是空闲的。

显存空间是这样分配的：每128个字节为1个完整轮次，第0个32字节映射到MC0并映射到某个显存颗粒，第一个对应MC1，第二个对应MC2，第三个对应MC3。传统的微机原理的数据扩展和地址扩展做不到这样，GPU内部通过逻辑门组合来选取。

L2 Cache

有一个好处就是：L2 Cache命中之后，就不用通过MC去GDDR中取数据了，极大地提升了带宽。

GPU的L2 Cache与CPU的L3 Cache比较像，都是LLC（Last Level Cache，末级缓存）。

Cache的具体详解参照相关技术文章和《CSAPP》（深入理解计算机系统）。

GPU的L2 Cache与CPU的L3 Cache对比

1，互联网络

CPU L3是通过Ring Bus互联在一起的。数据在环形总线上不断流转，数据走到对应的核心处则命中。这个在CPU核心数目较少的时候非常好，延迟极低。当核心数量几十个的时候，延迟会剧烈抖动（比如Intel就用的Ring Bus）。当然AMD不一样，用的IF Bus，这也是AMD多核性能暴打Intel的原因之一。

假设一个8核心的CPU，32MB的L3 Cache。那么32MB的L3 Cache会被划分为8个4MB的L3 Cache Slice。每个核心通过对地址进行计算，找到这个地址所在的L3 Cache块，然后发送请求到Ring Bus上取数据。

GPU的互联网络是Crossbar，24个SM可能同时向L2 Cache发送成百上千个内存读写请求。Crossbar是一个M×N的互联网络，M个输入源，N个输出源。只要两个SM请求的不是同一个L2切片，那么Crossbar的内部开关网络可以直接为这24个SM设置专属的物理直达通道。

梳理流程，以RTX4060为例：4060被划分为了3个巨大的GPC（Graphics Processing Cluster，图形处理集群），每个GPC内部有4个TPC（Texture Processing Cluster，纹理处理集群），每个TPC内部有2个SM，总共是24个SM处理器。一个SM内部包含（一个L0 i-Cache + 1个Warp Scheduler + 1个Dispatch（32 thread/clk）+ 1个16384×32bit的Register File + 1个FP32/INT32 + 1个FP32 + 1个ADA 4代Tensor Core + 4个LD/ST + 1个SFU）×4 + 1个共享的128KB L1 Data Cache/Shared Memory + 4个Tex + 1个3代RT Core。

1个GPC内部的SM产生的访存请求首先会在GPC内部的网络中进行第一轮的汇总排队，然后Global Crossbar将连接SM与对应的L2 Cache。Crossbar的Arbiter通过计算对应的SM与L2 Slice的对应关系，得到SM0和SM12都要去读取L2 Slice 2，这时候就导致了Bank Conflict。此时Arbiter根据Warp的年龄、请求的类型决定请求顺序，被Arbiter拦住的请求会丢到L2门口的MSHR（Missing Status Hold Register，缺失状态保持寄存器），SM不会死锁，而是执行其他的线程，直到数据返回。

2，缓存一致性

CPU的L3 Cache的缓存一致性要求非常高，假如核心1修改了变量x，核心2要用变量x，那么CPU会通过复杂的MESI协议发送侦听信号，让核心2知道数据更新，必须来L3拿新的。

GPU的L2 Cache要求极低，它放弃了全局的硬件一致性，L1缓存不保证同步。如果有好几个SM要抢同一个数据，那么必须显示调用atomicAdd或者threadfence。此时所有的同步逻辑会扔给L2 Cache里的特殊的ALU做同步处理。

3，智能预取

CPU L3内部有复杂的硬件预取器，如果是在连续读内存的场景下，比如读取0~30的数据时，它会提前预取40与50。

GPU L2则不会进行预取。

MSHR（Miss Status Holding Register）详细工作原理

MSHR是GPU缓存系统中用于处理缓存缺失的关键组件。当L2 Cache发生缓存未命中时，MSHR会记录请求信息并管理缺失数据的获取过程。

MSHR的工作流程：

  1. 当请求到达L2 Cache发现未命中时，MSHR会首先检查该地址是否已经有其他请求正在处理中
  2. 如果该地址正在被处理（其他Warp也在请求相同数据），MSHR会将这个请求合并到现有的请求条目中
  3. 如果是新请求，MSHR会分配一个新的条目，并开始向显存控制器发起数据获取请求
  4. SM可以继续执行其他就绪的Warp，而不需要等待数据返回，这就是”非阻塞缓存“的核心机制
  5. 当数据从显存返回时，MSHR会根据记录的请求列表，将数据分发给所有等待的请求者

MSHR的并发处理能力：

  • MSHR可以同时跟踪多个缓存缺失请求
  • 当多个Warp请求相同数据时，MSHR会合并这些请求，只向显存发起一次读取
  • 这种合并机制大大减少了显存带宽的浪费

MSHR与死锁避免：

由于GPU允许多个Warp同时运行，当一个Warp等待数据时，SM可以切换到其他Warp继续执行。MSHR的设计确保了这种切换不会导致系统死锁，只要MSHR还有空闲条目，系统就可以继续处理新的请求。

PCIe（Peripheral Component Interconnect Express）

PCIe挂在RingBus上，和L3 Cache、Memory Controller同级。

DDIO技术

在具体介绍DDIO之前，必须先了解哪里的数据是最新的：答案是L3 Cache里面的数据是最新的，因为L1、L2、L3 Cache的数据都在芯片内部，都受MESI协议掌控。

（1）如果CPU0写入x=1，存在L1中，PCIe需要读取L3 Cache的数据会先MESI一下，从而保证L3 Cache的数据始终最新。

（2）如果PCIe写入数据到L3 Cache的x中，但是CPU的L1 Cache中有x的记录，那么MESI会强制作废L1中的x，而从L3读取。

（3）MESI是自动执行的，硬件执行的，所以不会给CPU带来负担并且效率极高，因此不会出现缓存不一致的情况。

（4）但是如果是在DDR中呢？比如CPU更新的数据x，但是PCIe或者DMA直接从DDR读取x传给别的设备，这个x很可能是旧的值。

（5）如果DMA或者PCIe把DDR中的x更新了，但是CPU读取x是读取的Cache中的，这就会导致读取到旧的值。

（6）本质上是DDR不在CPU内部，不受MESI协议约束，所以很容易出现缓存不一致情况，所以要经常刷新缓存，而且数据流会经过Memory Controller，还要控制时序并且DDR负担会很大，所以开销很高。

（7）这就是DDIO引出的原因。

为什么Xeon的PCIe在L3 Cache内侧呢？其实这个图画的不太准，这个画的是逻辑图而不是物理布局图。实际上PCIe挂在RingBus上，与L3 Cache和Memory Controller同级。DDIO允许在L3 Cache内部划分一部分区域留给DDIO。因为Xeon作为服务器CPU，有着非常大的L3 Cache，最高应该是几百MB，所以留出来几十MB区域用于DDIO。

为什么只有Xeon有DDIO呢？因为Xeon的L3 Cache非常大，有一些空间留给DDIO，但是绝大部分情况都不会用到DDIO。因为Cache空间有限，如果稍微传递大点的包，则会导致Cache空间不足，Cache驱逐了其他的Cache记录，导致L3 Cache miss增大。

所以大多数情况还是老老实实用DDR做缓冲，而不是用少的可怜的L3 Cache。

PCIe的特性

1，点对点交换

以前的PCI（Peripheral Component Interconnect）总线是一个真正的总线。所有的设备都挂在同一个线上，同一个时刻只有一个设备说话，共享极其可怜带宽。

但是PCIe则不是，它采用了端到端的交换式架构。Root Complex（根节点）是CPU的内部控制器，相当于整个系统的高速枢纽。使用switch交换机进行路由，这样就可以多个链路并行通信不阻塞，比如GPU与内存的通信完全不影响网卡向CPU发送数据包。

2，物理通路

PCIe x 16代表16个Lane（通道），PCIe x 4代表4个Lane。

每个Lane走串行通道，每个串行通道包括一个Tx+、Tx-、RX+与RX-，串行全双工差分信号。并且x16表示16个Lane绑在一起，数据发送时会把数据差分从16个Lane上同时发送，接收端组装，从而实现了带宽倍增。

3，协议分层

PCIe像以太网一样采用分层协议：

  • 事务层：负责生成和解析TLP（Transaction Layer Packet，事务层数据包）
  • 数据链路层：给TLP加上序号包与CRC校验码，如果接收端校验错误会发起重传
  • 物理层：把组装好的包进行8b/10b或者128b/130b编码，防止信号全是0或者全是1导致失去时钟同步，并且转换成电信号传播

4，BAR（Base Address Register）映射

系统开机时，操作系统会读取GPU等设备的配置空间，并且把设备的内存显存映射到CPU的全局物理地址空间中。CPU写入某个特定的物理内存地址，实际上是通过PCIe写到了GPU的显存中。

5，DMA

GPU通过PCIe总线拥有了主控权，GPU的DMA自己可以生成PCIe读取请求，绕过CPU直接从系统的DDR内存把数据读取到显存中。DMA和CPU共用同一个Memory Controller。

6，通道自适应

如果CPU的PCIe通道数不够了，那么比如显卡占了16，SSD占了8，总共24。如果再来一个显卡，那么可能一个显卡就只分配到8个通道了。

GigaThread Engine

GigaThread Engine夹在PCIe接口和底层的运算集群（GPC/SM）之间，外部发来的所有任务必须经过它的拆解和分发，才能真正被执行。

1，拦截指令，接管全局

CPU发起GPU Kernel的时候，会通过PCIe总线发送一个Kernel Launch命令。GigaThread Engine会把CPU传过来的任务接过来并建立任务队列，此时由GPU接管任务。

2，资源分发

GigaThread Engine会扫描此任务的所需资源，比如里面有256个线程，需要10KB的Shared Memory和8000个寄存器，此时GigaThread Engine会瞬间扫描24个SM的实时状态，根据不同SM剩余的物理资源，决定把Thread Block派发给谁。

3，任务发送

一旦确定了要发送任务给哪些SM，GigaThread Engine就分发指令了。在CUDA中，任务被划分为一个巨大的Grid，每个Grid包含了成百上千个Block。GigaThread Engine会把这些Block像流水线一样分发给底层的SM，如果SM02运转极快，率先算完了分到的Block，那么GigaThread Engine会立刻察觉到，马上从队列中抽取一个新的Block给SM02。如果队列中没有Block了，那么GigaThread Engine就不分配任务了。

GigaThread主要从队列中分配Block，以Block为单位，而不是以Thread为单位，因为Block是硬件资源分配的原子单位。同一个Block的线程可以读写同一块Shared Memory，并且可以使用线程同步在某个节点互相等待。

Hyper-Q与CUDA Stream

自Kepler架构以来，Hyper-Q在GigaThread Engine里面放了32个硬件调度队列，每个硬件调度队列里面的执行顺序保持一致。并且发往SM的Block都携带了它所属流的信息，SM内部的硬件调度器确保这些不同的流的Block在逻辑上并不干扰。

CUDA Stream的概念：CUDA Stream是GPU上执行的一系列命令的有序队列。每个kernel调用和数据传输都可以指定在某个Stream中执行。不同Stream之间的操作可以并行执行，除非它们之间存在依赖关系（通过cudaStreamWaitEvent指定）。

Hyper-Q的价值：在没有Hyper-Q之前，即使代码中创建了多个CUDA Stream，由于硬件只支持一个硬件队列，这些Stream实际上会被串行执行。Hyper-Q允许不同的Stream真正并行运行，充分利用GPU的并行能力。

实际应用场景：

  • 计算与数据传输重叠：在Stream 1中执行kernel，同时在Stream 2中进行数据拷贝，让计算和数据传输同时进行
  • 多kernel并行：当一个kernel的计算强度较低时，可以让另一个更耗时的kernel同时运行
  • 图神经网络训练：可以使用多个Stream分别处理不同的图采样批次

使用示例：

OFA（Optical Flow Accelerator，光流加速器）

在图像处理中，光流指的是相邻两帧图像之间，像素点的运动矢量。比如第一帧有个球在(x, y)，第二帧运动到了(x+Δx, y+Δy)，那么光流就是把这个(Δx, Δy)找出来。这个计算很吃力，因为要对比两张图中每个像素块的相似度，如果用CUDA Core来算将会占用大量的浮点运算资源，速度慢。

OFA是一个独立的模块，用这个模块可以用来做帧生成，通过OFA快速识别出两帧游戏画面之间的物体运动轨迹，然后用AI强行插进去一帧中间画面。

DLSS与OFA的关系

DLSS（Deep Learning Super Sampling，深度学习超级采样）是NVIDIA的AI渲染技术的核心。DLSS使用深度学习模型来提升游戏画质和帧率，其中OFA产生的光流数据是DLSS 3的关键输入之一。

DLSS的工作原理：

  1. 当前帧：DLSS首先以较低分辨率渲染游戏画面
  2. 光流提取：OFA分析当前帧与前一帧的差异，提取出每个像素的运动矢量
  3. AI插帧：DLSS利用光流数据和时序信息，预测并生成高质量的中间帧
  4. 输出：游戏显示的是原本需要更高算力才能渲染出的高质量画面

DLSS 3的帧生成技术：

DLSS 3利用OFA产生的光流，结合游戏引擎提供的运动矢量数据，可以在已有的帧之间插入一个完整的AI生成帧。这意味着在相同硬件条件下，帧率可以提升近一倍。例如，原本在4K分辨率下只能跑60FPS的游戏，开启DLSS 3帧生成后可以达到接近120FPS的流畅体验。

技术限制：

  • OFA产生的光流图有固定的分辨率限制，当物体运动速度过快时，光流数据可能不准确
  • AI插帧的流畅度依赖于光流数据的质量，如果光流估计错误，插出的帧可能出现伪影
  • DLSS帧生成需要游戏引擎提供额外的运动矢量数据，不是所有游戏都支持

NVENC（NVIDIA Encoder）与NVDEC（NVIDIA Decoder）

用于将原始的像素数据压缩成视频格式，如H.264、HEVC以及AV1。把图像压缩成视频并且极大的减少数据量。如果使用GPU处理图像，那么使用NVENC打包成视频将会极大的降低数据占用率。

视频编解码流程概述

H.264/HEVC编码流程：

  1. 帧内/帧间预测：编码器分析当前帧与参考帧的差异，进行运动估计和运动补偿。对于Intra帧，使用帧内预测（利用当前帧内相邻像素预测）；对于Inter帧，使用帧间预测（利用参考帧中最佳匹配块预测）
  2. 变换与量化：将预测残差从空域转换到频域（DCT，离散余弦变换），然后进行量化以减少数据量
  3. 熵编码：使用CABAC（上下文自适应二进制算术编码）或CAVLC（上下文自适应可变长编码）对量化后的系数进行编码，进一步压缩数据
  4. 去块滤波与SAO：解码端会使用去块滤波器消除块效应，并使用SAO（样本自适应偏移）改善画质
  5. 环路滤波：在HEVC中还有更复杂的环路滤波器来提升视频质量

NVENC/NVDEC的硬件加速：这些硬件单元专门优化了上述流程中最耗时的部分：

  • 运动估计（ME）：专用硬件可以快速完成光流和运动矢量计算
  • 运动补偿（MC）：专门的硬件电路可以快速完成插值和像素混合
  • 变换量化（TQ）：专用硬件可以快速完成DCT和逆DCT操作
  • 熵编码/解码：专用硬件实现CABAC/CAVLC的高速编解码

GPC（Graphics Processing Cluster，图形处理集群）

GPC是用来分块管理的，如果顶层的GigaThread Engine一次去管理所有线程，那么效率将会非常低，所以需要分层自治。GPC、TPC都是这样的，并且分模块了之后，后面GPU也便于加资源与减资源。

并且一个GPC坏了不影响其他GPC工作。

Raster Engine（光栅化引擎）

光栅化：在计算机图形学中，所有的三维物体都是由无数个三角形组成的。输入3D空间中的三个顶点坐标(x, y, z)，输出这个三角形覆盖屏幕上的哪些像素点就可以了。

输出的所有像素点会被打包成Warp，交给下方的SM，然后交给SM内部的像素着色器算出具体的颜色就可以了。

为什么3个GPC各自有一个Raster Engine，这样可以负载均衡。如果有3个Raster Engine，那么可以把屏幕分为三块，3个GPC一个负责画左边，一个负责画中间，一个负责右边，这样就可以让三个光栅化引擎同时开工，保证渲染帧率。

TPC（Texture Processing Cluster，纹理处理集群）

TPC不是为了增加计算核心而存在的，这是为了资源共享和几何处理。如果只放一个SM，控制逻辑的开销太高；如果放四个，内部数据交换的延迟又太大。2个SM是目前物理设计上的”黄金比例”。

SM（Stream Multiprocessor，流多处理器）

SM里面有很多资源，分别解释如下。

L0 i-Cache + Warp Scheduler + Dispatch

L0 i-Cache是存储已经解码好的、随时准备发送的指令，访存极快。如果每次都去L1甚至显存里面找指令，延迟会非常高。

为什么只有L0 i-Cache而没有L0 d-Cache呢？因为L0 iCache是高度一致且共享的，但是数据是个性化而且海量的，所以只有L0 iCache。其实寄存器可以当作数据的L0 Cache，访问延迟几乎为0。几乎所有的SIMT架构，都会重视i Cache，D Cache很难。L0 iCache的任务是用最短的路径将指令广播给这个Warp，也就是32个核心。

如何实现广播的呢？一个L0 iCache连接了所有的计算核心，通过指令发射总线传递指令并进行广播，每个CUDA Core在主干道上监听信号即可完成广播操作，一次发送完成，而不是32次独立的发送。

Warp Scheduler是线程束调度器，每个调度器管理一组Warp，它盯着几十个Warp，必须在每个时钟周期做出决定。

每个SM子块都有一个Warp Scheduler。Warp Scheduler会把这几十个Warp在一个时钟周期内分为两类：

（1）非就绪态：等待内存数据，等待上一个数据计算完成，等待同步（__syncthreads）。

（2）就绪态：随时准备进入核心计算的。

所以如果就绪的Warp太少了，那么流水线就会产生停顿了。

然后如果就绪态任务有好几个，调度器通常使用轮询调度（Round Robin）或者贪婪最老策略：

  • 贪婪：如果一个Warp正在跑，且下一条指令也准备好了，就一直跑下去，能提高L0 Cache命中率
  • 最老：如果一个Warp被阻塞了，那么就选在队列里等待最久的Warp。这种切换是0开销的，因为每个线程的寄存器是在物理上分配好的，切换时只需要一个”指针”，瞬间完成

补充：CPU切换进程需要切换沉重的上下文，将所有的旧现场（寄存器的值、PC、栈指针等统统保存到内存），然后从内存中读取下一个线程之前存好的数据，重新填满寄存器，然后Cache都会失效，都要重新从内存中读取数据。

一旦选中了某个就绪的Warp，调度器将会发出指令，从L0 iCache中获取对应的指令，交给Dispatch Unit，然后Dispatch通过广播线，在一个周期内把指令压入32个核心。

Dispatch分发单元：调度器选好了Warp，Dispatch负责把指令分发下去。分发单元可以在一个时钟周期内，把一条指令同时分发给32个线程，即一个Warp，然后把指令发给对应的执行单元，如CUDA Core或者Tensor Core。

Register File

一个SM有4个Register File，总共16K×4 = 64K个寄存器，然后1个寄存器32bit，4字节。所以一个SM有256KB个寄存器容量。

这个寄存器不是排列成一个长条，而是分布在不同的存储体中。只要32个线程访问的寄存器地址不产生Bank冲突，数据就能瞬间就位。

如果核函数寄存器占用太多，比如一个函数定义了64个局部变量，占用64个寄存器。一个Warp有32个线程，消耗32×64 = 2048个寄存器。而一个SM有65536个寄存器，那么65536/2048 = 32。也就是这个SM最多同时容纳32个Warp。如果Kernel更复杂，每个线程用了128个寄存器，那么一个SM只能跑16个Warp，如果这几个Warp都在等显存数据，那么GPU就会空转。如果算法需要的寄存器超过了硬件限制，通常单个线程上限是255个，那么装不下的变量会偷偷挪到Local Memory中。Local Memory实际存在显存里，原本一个周期就能读到的变量，现在要跑400个周期。所以可以通过__launch_bounds__来让编译器限制寄存器的个数，变量用完就尽量释放或者复用同一个变量名，让编译器能更聪明的回收寄存器。并且中间计算结果如果很多线程都要用，那么存到Shared Memory中要划算的多。

为什么寄存器是私有的，但是访问也有可能触发Bank Conflict呢？因为寄存器堆是按照Bank排列的，如果一个线程读取的数据，比如执行c=a+b，a与b在同一个bank，那么就会触发bank conflict，因为一个Bank通常只有1个读写端口，必须要先读a再读b，从而效率降低了。

但是现在编译器非常智能，它会自动调整寄存器的分配顺序，尽可能通过物理布局避免冲突。

SM分配资源的最小单位是Block，一个Block必须完整地塞进SM，绝对不能被拆分。一个SM的极限是48个Warp，1536个线程。通过Nsight Compute发现一个线程占用33个寄存器，那么假设8倍取整按照40个来算。1536×40 = 61440 < 65536，所以寄存器非常充足。如果一个Block有1024个线程，那么一个Block只能放1个Block，剩下的空间都被浪费了，所以说Block最好是256或者512。

如果想算理论占有率，活跃Warp数量 = min（W_max, W_block, W_reg, W_smem）：

  • W_max是理论最高一个SM能并行多少个Warp，4060是48对应1536个线程
  • W_block是通过Block配置来算，比如Block为1024时，1个SM最多放1个Warp也就是1024个线程也就是32个Warp
  • W_reg是寄存器限制，通过Nsight Compute看一下一个线程占用多少寄存器，比如一个线程占用64个寄存器，那么65536/64 = 1024，最多1024个线程
  • Shared Memory也是一样的，比如每个Block申请了40KB的Shared Memory，那么对于128KB的Shared Memory大小来说，最多就能存放3个Block

并且由于Shared Memory和L1 DataCache共用一个内存区域，就会导致L1 DataCache容量减小。

CUDA编程实践建议

基于上述硬件特性，以下是一些实用的CUDA编程建议：

1. Block大小选择：

  • 对于计算密集型kernel，256或512线程的Block通常能取得较好的效果
  • 避免使用过大的Block（如1024线程），因为会导致资源利用不充分
  • 如果kernel需要使用大量Shared Memory，考虑减少每个Block的线程数以容纳更多Block

2. 避免Shared Memory Bank Conflict：

  • 将Shared Memory按列访问模式设计：即tile[row][col]，但按tile[col][row]的方式访问一列
  • 使用填充列（padding）使数据错开Bank边界
  • 使用__syncthreads()前确保所有线程都已完成对Shared Memory的写入

3. 利用warp级原语优化：

  • 使用__shfl()进行warp内的数据交换，比使用Shared Memory更快
  • 使用__ballot()、__popc()等warp级原语进行并行规约
  • 利用warp级别的原子操作减少锁竞争

4. 合并内存访问：

  • 让线程访问连续的内存地址，使一个warp的32个线程请求可以合并
  • 注意数据在内存中的布局，避免Stride大于1的访问模式

5. 充分利用硬件并行性：

  • 创建多个CUDA Stream让计算和数据传输重叠
  • 使用异步内存拷贝（cudaMemcpyAsync）配合Stream使用
  • 合理使用事件（cudaEvent）来同步Stream间的依赖

6. 使用Nsight分析：

  • 检查GPU利用率，找到瓶颈所在
  • 分析内存带宽使用情况
  • 检查warp调度效率

FP32/INT32与FP32

一个块有16个FP32/INT32计算单元与16个FP32计算单元，加起来就是128个CUDA cores。如果纯计算FP32，那么这128个CUDA核心都可以用于计算，如果只用于计算INT32，那么只有64个INT32核心。

那么假设FP32和INT32需要同时计算，那么核心只有16个但是线程有32个，那么硬件会花2个时钟周期来执行这个Warp。

FP32：FMA、FADD、FMUL、FMNMX、FCP、F2F，但是不能计算sin、cos（没有sin和cos指令，除非分为多个汇编指令来近似计算）。

INT32：ISCADD、LEA、IADD、IMUL、ISETP、SHF、BFE等。

SFU（Special Function Unit，特殊函数单元）

数量极其稀少，一个SM子块中只有4个SFU计算单元，吞吐量极小。

专门用于计算特殊函数：

  • __sinf(x)、__cosf(x)以及__sincosf正余弦计算
  • __exp2f、__log2f、__expf、__logf、__log10f对数计算
  • RCP倒数计算、平方根倒数

ADA 4th Generation Tensor Core（张量核心）

Tensor Core张量核心是专门用于计算硬件矩阵乘加的。比如4×4的矩阵A与B以及累加矩阵C，那么D = A×B + C瞬间会完成64次乘法与加法运算，这个操作被称为MMA（Matrix Multiply-Accumulate，矩阵乘累加）。

Tensor Core支持FP8，以及对0的压缩计算。

虽然每个SM子块只有4个Tensor Core，但是它吞吐量是矩阵级别的，可以贡献80%以上的浮点运算理论峰值。除非把运算转换成矩阵运算，否则基本用不上张量计算单元。

Transformer Engine与FP8加速

Transformer Engine是NVIDIA提供的库，专门用于加速Transformer模型的训练和推理。它充分利用了Tensor Core的矩阵运算能力，并支持FP8精度来进一步提升性能。

FP8的优势：

  • 相比FP16，FP8可以提供2倍的计算吞吐量和更低的内存占用
  • 虽然精度较低，但Transformer模型对精度损失相对不敏感
  • NVIDIA H100的Tensor Core原生支持FP8，可以在硬件层面高效执行

Transformer Engine的自动精度选择：

  • 提供”自动混合精度”模式，会自动在FP32、FP16、BF16、FP8之间选择最优精度
  • 识别模型中哪些层可以安全地使用低精度，哪些层需要保持高精度
  • 最大化性能的同时尽量减少精度损失对模型收敛的影响

LD/ST（Load/Store Unit，加载/存储单元）

由于所有的计算核心FP32、INT32、SFU、Tensor Core都只能操作寄存器，如果数据在显存里面，那么这些计算核心无法获取。

一个SM子块有4个LD/ST，那么一个SM有16个LD/ST单元。

LD/ST和INT32计算单元结合非常紧密，因为INT32通常用来算地址，INT32就会把这些地址交给LD/ST单元。

然后LDG.E指令从全局内存读取数据，然后STG.E把寄存器放回到全局内存中。注意LD/ST只用来获取数据，不用来获取指令，Warp Scheduler有自己的取值单元，它会直接去L0 i-Cache中获取指令，找不到就去L1 i-Cache，然后去L2 Cache，再不行就去显存中找了。

当Warp Scheduler发出一条获取显存的指令时，Warp里的32个线程会同时向LD/ST单元提交32个内存地址，然后LD/ST会判断这32个地址能不能打包，也就是合并访问。

LD/ST会依次通过L1 Cache和L2 Cache，在哪一层找到就直接获取。

L1 Data Cache / Shared Memory

L1 D-Cache和Shared Memory被放在同一内存区域，共同瓜分128KB的空间。

Shared Memory是软件托管的，必须自己手动申请。

L1 D-Cache是GPU管理的，无法自己手动管理。

Shared Memory和L1 D-Cache的大小可以手动划分。

L1 D-cache的操作单位是一个缓存行。

由于Shared Memory可以自己手动控制，所以一般的场景下，也可以用Shared Memory代替L1 Cache，从而便于更精细的管理。

总结一下Cache的流程：

  • 指令流：L0 i-Cache（SM子块私有）-> L1 i-Cache（SM共享）-> L2 Cache（全局混合）-> 显存
  • 数据流：L1 D-Cache/Shared Memory（SM共享）-> L2 Cache（全局混合）-> 显存

为什么没有L0 D-Cache？因为数据访问太暴力了，并且数据不能广播，所以会导致没有L0 D-Cache，只有L0 i-Cache。

Shared Memory

距离计算核心非常近，访问只需要20~30个时钟周期，带宽可以达到恐怖的几TB/s，范围是以Block进行隔离的。同一个Block里的线程可以互相看到并读写这个Shared Memory中的数据。不同的Block无法互相看到彼此的Shared Memory。

但是Shared Memory搭配使用通常需要__syncthreads()，因为同一个Block里的线程有快有慢，只有用__syncthreads进行同步之后，才能保证Shared Memory里的数据是正确的。

但是要注意，Shared Memory也是基于Bank存储的，如果32个线程同时读取了Shared Memory的同一个Bank的不同地址，由于同一个Bank读取端口只有1，32个线程只能排队访问，又再一次触发了Bank Conflict。Shared Memory有32个Bank，每个Bank 4字节，所以在使用共享内存的时候，一定要注意不要触发Bank Conflict。

Crossbar

Shared Memory有广播功能，比如一个Block里的Warp都去读tile[0]，那么只需要1次读取就可以了，而不是需要32次读取。Shared Memory为什么有广播功能？答案是通过Crossbar Switch（交叉开关网络）。Crossbar Switch有数据路由与复制的能力，而且还有多播能力。

只要有Crossbar Switch，广播、多播、合并相同请求都可以实现。

  • L0 iCache有广播能力，但没有Crossbar，因为它总是广播，直接做了一个分发树或者叫信号扇出网络
  • 常量内存：也是典型的一对多的关系，也是只有一个分发树
  • **L1 D-Cache/Shared Memory**：多对多，用了Crossbar
  • **L2 Cache**：也有Crossbar，而拥有广播和多播的能力

Tex纹理内存

普通的LD/ST读取内存时，喜欢的是一维连续（行优先），如果要读取一个像素点与其下方的点，那么缓存命中率可能极低。Tex在内部对数据做了齐次映射，把2D的矩阵重新排列成一个小方块。但是纹理内存主要是只读。

齐次映射

更通用的说法是瓦片化（Tiling）或者Swizzling（重组），目的是保证空间均匀性，保证无论是沿着X方向读取还是Y方向读取，内存访问的开销是均匀对称的。

比如一个点的坐标是(x, y)，x = 01（二进制）y = 10（二进制），取y的bit0与x的bit0组成01，取y的bit1与x的bit1组成10，合起来就是1001，这个就是Z-order曲线映射，而且不需要知道原始二维矩阵的尺寸就可以做。下面是Z-order的映射模拟代码：

纹理内存内部还自带了专用的硬件插值电路，从里面直接读取数据的时候，读出来的就可以是插值后的结果。

假如(0,0)=100，(1,0)=200，(0,1)=150，(1,1)=250。如果想要获取坐标为（0.4, 0.6）处的坐标值，那么首先计算y=0时，x=0.4：100 + (200-100)×0.4 = 140；y=1时，x=0.4：150+(250-150)×0.4 = 190。

然后140 + (190-140)×0.6 = 170，最终纹理单元会返回170。

纹理内存里面有专门的硬件电路，用于在一个周期内抓取出来4个数，然后使用一个乘加电路来进行计算，不占用其他计算单元的资源。

RT Core 3rd Generation

主要是用于计算光线的，就是光线与三角形求交。判断光线与物体的相交，实现更高的几何细节等等，提高着色器（决定像素颜色）的执行效率。

光线追踪技术原理

光线追踪（Ray Tracing）是一种渲染技术，通过模拟光线在场景中的传播路径来生成逼真的图像。RT Core专门加速了光线追踪中最耗时的部分。

光线追踪的基本流程：

  1. 光线生成：从摄影机位置发射射线，穿过屏幕上的每个像素
  2. 场景求交：射线与场景中的三角形求交，找到最近的交点。这是最耗时的部分，RT Core通过专用硬件加速
  3. 阴影计算：从交点向光源发射阴影射线，检测是否有物体遮挡
  4. 材质属性：根据交点的材质属性（如金属、漫反射、镜面反射等）计算反射和折射
  5. 光线递归：对于反射和折射光线，重复步骤2-4，直到达到最大递归深度或光线强度衰减到阈值以下

RT Core的硬件加速：

  • 专用求交电路：RT Core内部有专门的Box和Triangle求交单元，可以在硬件层面并行执行求交计算
  • BVH遍历加速：场景通常使用BVH（Bounding Volume Hierarchy，层次包围盒）组织三角形，RT Core有专用硬件加速BVH遍历
  • 过滤加速：RT Core支持Motion Allow、Primitive Clip Culling等优化，减少需要求交的物体数量

实用建议：

  • 使用纹理存储三角形顶点属性，减少顶点数据的访问开销
  • 对于动态场景，使用RTXDI（ReSTIR GI）等全局光照技术替代纯光线追踪
  • 利用Tensor Core加速降噪：实时光线追踪的画面噪点可以通过AI降噪器快速去除。
